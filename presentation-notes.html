Model - Proxies and Data Objects

  • Data objects can be simple data holders
    • e.g. ArrayCollection, XMLListCollection, Value Object, etc.
  • Proxies control access to Data Objects by exposing an API
    • A UserProxy might have methods like saveUser(), deleteUser(), loadUsers()
  • Proxies may interact with remote services
  • Why have service interaction in the Model?
    • Encapsulation and portability
    • No explosion of classes for every action (e.g. Commands)
  • Proxies may broadcast but do not listen for notifications
    • No dependencies on view or controller layer
    • Portability (desktop, mobile, modules)

View - Mediators and View Components

  • View Component is any UI component
    • e.g. DataGrid, List, or any custom component
  • Mediators look after View Components
    • Listen for dispatched events
    • Inspect component properties and methods
  • Mediators interact with system on behalf of View Component
    • View Component has no knowledge of the framework
    • View Component should encapsulate its own behavior
      • e.g. form validation, transition management, subcomponent visibility
  • Mediators may broadcast and listen for notifications

Controller - Commands

  • Contain business logic of the application
  • Stateless and created only when needed
  • Coordinate complex activities like startup and shutdown
  • Useful for actions that can be triggered from multiple places
  • Expect to write few commands!
    • Mediator performs critical decoupling of View Component from Model
    • Service interaction is moved to the model
  • Commands may broadcast and are triggered by notifications

Facade

  • Simple API for retrieval, registration and removal of core actors
  • Hides the Model, View and Controller singletons
  • Any Proxy, Mediator or Command has access to Facade
    • Via an inherited "facade" property
  • Create a concrete Facade called ApplicationFacade
    • Handles registration of commands
    • Defines application-level notification constants
